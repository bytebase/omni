# Oracle Parser Compatibility Report And Gap Map

Date: 2026-05-15
Last updated: 2026-05-27

Goal: produce and maintain a compatibility report and gap map for the Oracle parser. This document separates measured local parser health from real Oracle compatibility evidence.

## Executive Read

The Oracle parser has strong local parser hygiene. Phase A core SQL, Phase B DDL, Phase C PL/SQL, Phase D keyword/identifier, and Phase E admin/catalog/unsafe parser gap passes are complete for the expanded Oracle Free reference manifest.

Measured local gates are green:

- Parser methods: 452/452 `parse*` methods return errors.
- Silent parser error discards: 0.
- Local corpus: 128 statements, 0 parse violations, 0 Loc violations, 0 crashes.
- Coverage gates: 62 soft-fail scenarios, 121 strictness scenarios, 344 keyword scenarios, 171 classified BNF rows, 251 Loc-node rows, 154 direct Loc fixtures.
- P2 manifest: 171/171 rows have positive, negative, and Loc test evidence.

The broader compatibility gap is still reference depth:

- `p2_coverage.tsv` has 171 rows, but all 171 currently use `local_structure_tests_only` as reference evidence.
- `reference_oracle.tsv` has only 33 rows: 9 accept, 18 reject, 6 unsafe.
- The new `compat_oracle.tsv` manifest has 608 rows, including 188 Phase A core SQL rows, 180 Phase B DDL rows, 80 Phase C PL/SQL rows, 80 Phase D keyword/identifier rows, and 80 Phase E admin/catalog/unsafe rows, and has been run against Oracle Free via `ORACLE_PARSER_REF_CONTAINER=1`.

Expanded compatibility result after the Phase E completion pass:

| Outcome | Rows |
|---|---:|
| Oracle/omni accept match | 361 |
| Oracle/omni reject match | 182 |
| omni too lenient | 0 |
| omni too strict | 0 |
| Oracle semantic/catalog noise | 2 |
| Oracle privilege/catalog noise | 45 |
| unsafe skipped | 18 |

Conclusion: the parser is well-instrumented and locally regression-tested. On the expanded Oracle-backed manifest, it has no remaining omni too-lenient or too-strict syntax mismatches across the 590 non-unsafe rows. The non-direct matches are explicitly classified catalog or privilege noise.

### 2. BNF Depth

Historical BNF coverage still shows the real grammar-depth debt:

| Status | Rows |
|---|---:|
| covered | 11 |
| partial | 86 |
| deferred | 47 |

Interpretation:

- `covered` means there is stronger parser-surface evidence.
- `partial` usually means representative local tests exist, but not exhaustive branch or Oracle-reference proof.
- `deferred` means specialized syntax is structurally tracked but not compatibility-measured.
- `catalog` means syntax is mixed with privilege, catalog, or instance state and needs a controlled reference setup.

### 3. Local P2 Versus Compatibility

`p2_coverage.tsv` is complete as a parser-structure manifest:

- 171 total rows.
- 171 `p2_done`.
- 171 rows with positive tests.
- 171 rows with negative tests.
- 171 rows with Loc tests.

But every row has `reference_evidence=local_structure_tests_only`. That is a useful parser engineering gate, not an Oracle compatibility proof.

Gap: add a second compatibility field or companion manifest that records `oracle_match`, `oracle_too_strict`, `oracle_too_lenient`, `oracle_unsafe`, `version_dependent`, and `not_run`.

### 4. Corpus Representativeness

It is good as a crash/Loc/parser-regression corpus. It is weak as a compatibility corpus because only 3 statements are expected failures and most statements are not verified against real Oracle in the default run.

Gap: split corpus roles explicitly:

- Regression corpus: no crash, no Loc violation, expected local parser behavior.
- Reference corpus: real Oracle accept/reject agreement.
- Unsafe catalog corpus: parse-only with setup/cleanup or privileged DSN.
