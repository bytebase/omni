# PG Parser FIRST-Set Consolidation Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Eliminate hand-maintained token-set duplication in omni's pg parser by introducing a single source of truth for each grammar production's FIRST set, validated against real PostgreSQL via testcontainer oracle.

**Architecture:** For each grammar production P, define one `isPStart()` predicate backed by a single `pLeadTokens` slice. Every lookahead probe in the parser calls `isPStart()` — no bare token `case` clusters allowed in probe positions. An AST-free oracle test runs each probe token against a real PG container using a production-specific SQL template, and asserts that omni's computed FIRST set matches PG's actual accepted set.

**Tech Stack:** Go, testcontainers-go (postgres:17-alpine — omni's pg parser targets PG 17 grammar per `pg/parser/type.go:9` and references PG 17 docs throughout; do not reuse `pg/catalog`'s PG 16 image since that predates omni's parser upgrades), pgx/v5 for error code classification.

---

## Background

### The triggering bug

A bytebase customer hit `syntax error at or near "json"` when omni tried to parse `CREATE FUNCTION f(data json) ...` generated by bytebase's `WalkThroughOmni` → `GetDatabaseDefinition` → omni catalog `Exec` pipeline.

Root cause chain:

1. `parseFuncArg` (omni/pg/parser/create_function.go:220) must disambiguate `param_name func_type` from bare `func_type`. After speculatively consuming a name, it peeks at the next token via `isBuiltinType()` (create_function.go:338).
2. `isBuiltinType()` is a **hand-written whitelist** of 17 tokens: `INT_P, INTEGER, SMALLINT, BIGINT, REAL, FLOAT_P, DOUBLE_P, DECIMAL_P, NUMERIC, BOOLEAN_P, BIT, CHAR_P, CHARACTER, VARCHAR, TIMESTAMP, TIME, INTERVAL`.
3. `JSON` is a `ColNameKeyword` in `keywords.go:236`, and `parseSimpleTypename` (type.go:170) handles it via `case JSON`. But `isBuiltinType()` is missing it.
4. Result: after consuming `data`, peek sees `json`, `isBuiltinType()` returns false, `isTypeFunctionName()` returns false (JSON is `ColNameKeyword`, not `TypeFuncNameKeyword`), parser falls to case 5, pushes `data` back as the type, and crashes on the next `json` token.

Also missing from the whitelist: `DEC`, `NATIONAL`, `NCHAR` — all `ColNameKeyword` types that `parseSimpleTypename` handles explicitly. (`BOOLEAN_P` is already present in the whitelist; the drift affected four tokens, not five.)

### Architectural root cause

omni's pg parser has a **structural asymmetry**:

- **Category layer is right.** `isColId` / `isColLabel` / `isTypeFunctionName` all consult `LookupKeyword(...).Category` against `keywords.go` — a single source of truth. Adding a new keyword in the right category automatically flows to every identifier check.
- **FIRST-set layer is wrong.** The parser also needs to ask "does this token start grammar production P?" (e.g., `SimpleTypename`, `func_type`, `a_expr`). These are the FIRST sets of productions, and omni has **no single source of truth** for them. Each probe point hand-writes its own token `case` cluster.

Currently the SimpleTypename FIRST set is duplicated in **three places**:

| Location | Purpose | State |
|---|---|---|
| `type.go:105 parseSimpleTypename` | Real SimpleTypename dispatch | ✅ Complete (20 tokens) |
| `expr.go:2997 isAExprConstTypeCast` | ConstTypename probe for `int '42'` cast | ✅ Complete (20 tokens) |
| `create_function.go:338 isBuiltinType` | func_arg disambiguation probe | ❌ Missing JSON, DEC, NATIONAL, NCHAR |

PG doesn't have this problem because bison's LALR(1) table is **automatically** derived from the grammar — FIRST sets are computed, not hand-maintained. omni is hand-written recursive descent, so it has to maintain FIRST sets explicitly, and "explicit" has become "duplicated".

### Why a single hand-maintained set isn't enough

Even if we collapse the three copies into one Go slice, future code will still drift: someone adds a `case XMLTYPE:` to `parseSimpleTypename` and forgets the slice. The only way to prevent this systemically is to make drift **fail loudly in CI**.

### Why PG testcontainer oracle is the right validator

We considered four validator designs:

1. **AST reflection** — test reads parser source via `go/ast`, compares switch cases to slice. Works, but oracle is "omni's own other file" — still self-referential. Doesn't catch category miscatalog. Each production needs bespoke AST traversal.
2. **Runtime dispatch table** — rewrite parse functions as `map[TokenType]parseFunc`. Single source of truth but heavily invasive; breaks stack traces and omni's recursive-descent idiom.
3. **Code generation** — generate both slice and switch from a DSL. Adds build infrastructure for a local refactor.
4. **PG testcontainer oracle** — probe real PG with production-specific SQL templates, classify errors by SQL state code, compare result set to omni's slice. Truly external oracle. One probe pattern scales to all productions. Matches omni's existing project-wide philosophy of "real DB as oracle" (catalog walkthrough, mysql/mssql oracle tests all do this).

We picked #4. The oracle being PG itself means:

- Drift between slice and `parseSimpleTypename` switch is caught (the thing that would catch JSON bug).
- Drift between omni and PG **grammar** is caught (future PG releases add a type → test reminds us).
- Category miscatalog is caught (if `bytea` were wrongly ColNameKeyword, the test would notice).
- One oracle pattern (`probeTemplate(tok) → sqlstate`) extends to every production without bespoke code.

## Goals and non-goals

### Goals

- Eliminate hand-maintained FIRST-set duplication in probe positions.
- For every grammar production with a FIRST-set probe, a single `isPStart()` function backed by a single slice.
- PG testcontainer oracle test that validates each slice against real PG.
- Code convention documented in `pg/parser/CLAUDE.md` preventing reintroduction.
- JSON bug is fixed — not as a standalone hotfix, but as the natural output of running the new oracle test on the existing code.

### Non-goals

- Not replacing category-based checks (`isColId` / `isColLabel` / `isTypeFunctionName`) — those already have a single source of truth.
- Not restructuring `parseSimpleTypename` into a dispatch table — switch stays.
- Not covering "reverse" probes like `isReservedForClause` (negative sets used in ambiguous contexts). Phase 3 audits whether any need the same treatment, but we don't rewrite them speculatively.
- Not adding static analyzer / linter — Phase 4 is documentation only; re-evaluate analyzer after 3 months of convention adherence.

---

## Phased plan

- **Phase 1** — Oracle infrastructure + SimpleTypename / Typename / ConstTypename (the JSON bug fix falls out of this)
- **Phase 2** — func_type + a_expr / c_expr FIRST sets (removes remaining known heuristics)
- **Phase 3** — Audit every multi-token `case` cluster across 12 parser files, migrate legitimate probes
- **Phase 4** — Document convention in `pg/parser/CLAUDE.md`, freeze pattern for future PRs

```go
package parser

import (
    "context"
    "database/sql"
    "errors"
    "fmt"
    "os"
    "strings"
    "sync"
    "testing"

    "github.com/jackc/pgx/v5/pgconn"

    "github.com/testcontainers/testcontainers-go"
    tcpg "github.com/testcontainers/testcontainers-go/modules/postgres"
    "github.com/testcontainers/testcontainers-go/wait"
)

type firstSetOracle struct {
    db  *sql.DB
    ctx context.Context
}

var (
    firstSetOracleOnce sync.Once
    firstSetOracleInst *firstSetOracle
)

// oracleSetupError remembers a setup failure so every subsequent test
// observes it and can fail or skip uniformly. sync.Once runs exactly
// once, so errors must be captured here rather than returned from Do.
var oracleSetupError error

func startFirstSetOracle(t *testing.T) *firstSetOracle {
    t.Helper()
    firstSetOracleOnce.Do(func() {
        ctx := context.Background()
        container, err := tcpg.Run(ctx, "postgres:17-alpine",
            tcpg.WithDatabase("omni_fs"),
            tcpg.WithUsername("postgres"),
            tcpg.WithPassword("test"),
            testcontainers.WithWaitStrategy(
                wait.ForLog("database system is ready to accept connections").
                    WithOccurrence(2)),
        )
        if err != nil {
            oracleSetupError = fmt.Errorf("container start: %w", err)
            return
        }
        connStr, err := container.ConnectionString(ctx, "sslmode=disable")
        if err != nil {
            _ = testcontainers.TerminateContainer(container)
            oracleSetupError = fmt.Errorf("conn string: %w", err)
            return
        }
        db, err := sql.Open("pgx", connStr)
        if err != nil {
            _ = testcontainers.TerminateContainer(container)
            oracleSetupError = fmt.Errorf("db open: %w", err)
            return
        }
        if err := db.PingContext(ctx); err != nil {
            db.Close()
            _ = testcontainers.TerminateContainer(container)
            oracleSetupError = fmt.Errorf("ping: %w", err)
            return
        }
        firstSetOracleInst = &firstSetOracle{db: db, ctx: ctx}
    })

    if oracleSetupError != nil {
        // CI must fail loudly — otherwise a misconfigured CI silently
        // skips every FIRST-set test and disables the entire guardrail.
        // Local dev without docker is allowed to skip.
        if isCI() {
            t.Fatalf("first-set oracle unavailable in CI: %v", oracleSetupError)
        }
        t.Skipf("first-set oracle unavailable (local dev): %v", oracleSetupError)
    }
    return firstSetOracleInst
}

// isCI reports whether we're running under continuous integration.
// Matches omni's existing conventions and major CI providers.
func isCI() bool {
    // Respect the standard CI env var set by GitHub Actions, CircleCI,
    // GitLab CI, etc. If omni has a project-specific variable, add it here.
    return os.Getenv("CI") == "true" || os.Getenv("CI") == "1"
}
```

**Step 2: Add probe result classifier**

```go
// probeResult classifies a PG error into accept/reject for FIRST-set purposes.
type probeResult int

const (
    probeAccept probeResult = iota // syntax OK, may or may not be semantically valid
    probeReject                    // syntax_error (42601)
)

func classifyProbeErr(err error) probeResult {
    if err == nil {
        return probeAccept
    }
    var pgErr *pgconn.PgError
    if errors.As(err, &pgErr) {
        if pgErr.Code == "42601" {
            return probeReject
        }
    }
    return probeAccept
}
```

**Step 3: Add the per-token probe runner with savepoint isolation**

```go
// probe runs one SQL statement inside a savepoint so side effects roll back.
func (o *firstSetOracle) probe(t *testing.T, sqlStr string) probeResult {
    t.Helper()
    tx, err := o.db.BeginTx(o.ctx, nil)
    if err != nil {
        t.Fatalf("begin: %v", err)
    }
    defer tx.Rollback()
    _, err = tx.ExecContext(o.ctx, sqlStr)
    return classifyProbeErr(err)
}
```

**Step 1: Enumerate candidate tokens — keywords AND non-keyword seeds**

The driver must iterate over every token class that can legally start a grammar production, not just keywords. Phase 1's productions (SimpleTypename/ConstTypename/Typename/func_type) happen to have keyword-only hard FIRST sets plus a `type_function_name` category path, so keyword iteration alone would cover them. Phase 2 adds `a_expr` whose FIRST set includes punctuation (`'('`, `'+'`, `'-'`), literal tokens (`ICONST`, `FCONST`, `SCONST`, `BCONST`, `XCONST`), and bare `IDENT`. We bake non-keyword candidates into the driver from day one so Phase 2 inherits it for free and Phase 1 tests double-check that the non-keyword paths don't introduce false positives.

```go
// candidateKind categorizes a probe token.
type candidateKind int

const (
    kindKeyword  candidateKind = iota // From the Keywords table
    kindIdent                          // Bare identifier, not a keyword
    kindLiteral                        // Numeric / string / bit / hex literal
    kindPunct                          // Operator or punctuation opener
)

// candidateToken represents one token to probe in a FIRST-set test.
//
// For keyword candidates, `name` is both the SQL surface form (lowercase
// keyword) and the string used to construct Parser.cur.Str, and `token`
// is the omni token constant from keywords.go.
//
// For non-keyword candidates, `probeSQL` is the SQL fragment to
// interpolate into the probe template (may include trailing disambiguators
// like `"(1)"`, `"+1"`), and `token` is the omni token constant for the
// LEADING token of that fragment — e.g. the `'('` rune for `"(1)"`, the
// PARAM constant for `"$1"`. When the predicate is called, the test
// constructs Parser.cur = Token{Type: token, Str: leadStr} so the
// lead-token check reflects what the first lexed token actually is.
type candidateToken struct {
    name     string            // SQL keyword string (keyword kind) OR leading-token literal (non-keyword)
    probeSQL string            // SQL fragment interpolated into the probe template
    display  string            // Human-readable label for test output
    token    int               // omni token type constant — MUST be set for every candidate
    category KeywordCategory   // Valid only when kind == kindKeyword
    kind     candidateKind
}

// For keyword candidates, probeSQL defaults to the keyword name (set in
// allCandidates via the renderer). For non-keyword candidates, probeSQL
// is defined explicitly below.
//
// nonKeywordCandidates is the fixed seed set of non-keyword starters
// covering all production FIRST sets in the omni pg parser.
//
// Every entry has a concrete `token` value so assertPredicateMatchesPG
// can call the predicate via onNonKeyword with a meaningful Parser.cur.
// Token constants like ICONST, FCONST, SCONST, BCONST, XCONST, PARAM,
// and IDENT are defined in tokens.go; punctuation leads use the Go rune
// value (PG parser's lexer emits '(' as int('(') etc.).
var nonKeywordCandidates = []candidateToken{
    {name: "omni_probe_ident", probeSQL: "omni_probe_ident", display: "IDENT",
     token: IDENT, kind: kindIdent},
    {name: "1", probeSQL: "1", display: "ICONST",
     token: ICONST, kind: kindLiteral},
    {name: "1.5", probeSQL: "1.5", display: "FCONST",
     token: FCONST, kind: kindLiteral},
    {name: "'x'", probeSQL: "'x'", display: "SCONST",
     token: SCONST, kind: kindLiteral},
    {name: "B'1'", probeSQL: "B'1'", display: "BCONST",
     token: BCONST, kind: kindLiteral},
    {name: "X'AB'", probeSQL: "X'AB'", display: "XCONST",
     token: XCONST, kind: kindLiteral},
    {name: "$1", probeSQL: "$1", display: "PARAM",
     token: PARAM, kind: kindLiteral},
    // Punctuation / unary operators. probeSQL uses a trailing operand
    // (e.g. "(1)", "+1") to make PG's overall statement parse when the
    // lead token is legal; the test only cares whether PG rejects with
    // 42601 before even reaching the operand. Token constant is the Go
    // rune literal, since omni's pg lexer emits them that way.
    {name: "(", probeSQL: "(1)", display: "LPAREN",
     token: int('('), kind: kindPunct},
    {name: "+", probeSQL: "+1", display: "PLUS",
     token: int('+'), kind: kindPunct},
    {name: "-", probeSQL: "-1", display: "MINUS",
     token: int('-'), kind: kindPunct},
}

// Verify in Task 1.2 Step 1 that PARAM / ICONST / etc. constants exist in
// `pg/parser/tokens.go`; if any are missing, add them to the plan's
// "known-unknowns" section and pick alternate candidates. omni already
// uses PARAM in expr.go per the parser source, so this should hold.

func allCandidates() []candidateToken {
    out := make([]candidateToken, 0, len(Keywords)+len(nonKeywordCandidates))
    for _, kw := range Keywords {
        out = append(out, candidateToken{
            name:     kw.Name,
            probeSQL: kw.Name, // keywords probe as bare keyword; renderer expands
            display:  kw.Name,
            token:    kw.Token,
            category: kw.Category,
            kind:     kindKeyword,
        })
    }
    out = append(out, nonKeywordCandidates...)
    return out
}
```

Per-production tests may filter to `kindKeyword` only (Phase 1 does this for SimpleTypename/Typename/ConstTypename, since non-keyword starters are covered by the category path and would produce meaningless SQL like `SELECT CAST(NULL AS 1)` that PG would reject for the wrong reason). Phase 2's `isAExprStart` consumes the full list. The per-test filter makes the scope explicit and visible in each test.

**Step 2: Build the generic production runner with per-candidate rendering**

The naive approach — substituting `c.name` into the probe template — fails for multi-token types. PG rejects `SELECT CAST(NULL AS double)` but accepts `... AS double precision`; it rejects `... AS national` but accepts `... AS national character`; it rejects `... AS character varying` bare but accepts `... AS character varying(1)`. If the oracle probes only the bare form, it marks `DOUBLE_P`, `NATIONAL`, etc. as rejected and pushes us to delete real lead tokens from the slice — which would cripple the parser.

The fix: each production-specific test provides a **render function** that produces one or more candidate forms per token, and the probe accepts if **any** form parses. This brute-force approach keeps the oracle honest without re-encoding per-token grammar rules.

```go
// renderFn produces one or more candidate renderings for a token.
// The runner accepts a token if ANY rendering parses under PG.
type renderFn func(c candidateToken) []string

// renderBare returns only the candidate's configured probeSQL. Use this
// for probes where the token must stand alone (e.g., ConstTypename
// 'literal') and no multi-word rendering is needed.
func renderBare(c candidateToken) []string { return []string{c.probeSQL} }

// renderTypeCandidate expands a token into the combinations that real
// PG type grammar requires: bare, parenthesized type modifiers, and the
// multi-word forms (PRECISION, CHARACTER, VARYING) that the PG lexer
// emits as separate tokens. Tokens that REQUIRE a trailing element (like
// SETOF, which must be followed by an inner type) get dedicated renderings
// that supply a valid continuation.
//
// Enumerating every legal form explicitly is intentional — any form that's
// missing here would make the oracle drop a real lead token. The list is
// derived from gram.y:14339..14760 (SimpleTypename and its immediate
// sub-productions) plus Typename's SETOF prefix (gram.y:14304).
func renderTypeCandidate(c candidateToken) []string {
    // For non-keyword candidates, use probeSQL directly — no multi-word
    // expansion applies. For keywords, expand into all plausible forms.
    if c.kind != kindKeyword {
        return []string{c.probeSQL}
    }
    base := c.name
    // Tokens that require a trailing inner type. SETOF is the only one in
    // Typename today; extend this list if Phase 3 finds more.
    if base == "setof" {
        return []string{
            "setof int",
            "setof varchar(10)",
            "setof json",
        }
    }
    return []string{
        base,
        base + "(1)",
        base + "(1, 1)",
        base + " precision",                // DOUBLE PRECISION
        base + " character",                // NATIONAL CHARACTER
        base + " character(1)",
        base + " varying",                  // BIT VARYING, CHARACTER VARYING
        base + " varying(1)",
        base + " character varying",        // NATIONAL CHARACTER VARYING
        base + " character varying(1)",
    }
}

// probeOutcome records the result for a single candidate: which token it
// was, and whether PG accepted any of its renderings.
type probeOutcome struct {
    candidate candidateToken
    accepted  bool
}

// runFirstSetProbe probes each filtered candidate via every rendering
// returned by `render`. A candidate is marked accepted if any rendering
// parses under PG. Each probe runs inside a savepoint for isolation.
//
// The return value is a slice of per-candidate outcomes, preserving one
// entry per candidate (including duplicates by token value — e.g. every
// non-keyword candidate has token == -1 but distinct displays). The
// assertion helper iterates this slice for bidirectional validation.
func runFirstSetProbe(
    t *testing.T,
    o *firstSetOracle,
    probeTemplate string, // fmt template with a single %s slot
    render renderFn,
    filter func(candidateToken) bool,
) []probeOutcome {
    t.Helper()
    var outcomes []probeOutcome
    for _, c := range allCandidates() {
        if !filter(c) {
            continue
        }
        accepted := false
        for _, form := range render(c) {
            sql := fmt.Sprintf(probeTemplate, form)
            if o.probe(t, sql) == probeAccept {
                accepted = true
                break
            }
        }
        outcomes = append(outcomes, probeOutcome{candidate: c, accepted: accepted})
    }
    return outcomes
}

// Filters for common cases.
func filterKeywordsOnly(c candidateToken) bool { return c.kind == kindKeyword }
func filterAll(c candidateToken) bool          { return true }
```

**Step 3: Add the bidirectional assertion helper**

These helpers are used by every production test starting from Task 1.3, so they must land in Task 1.2 — if they were added later, Task 1.3's first test would fail to compile for missing symbols rather than for the intended missing FIRST-set definition.

```go
// predicateProbe dispatches a FIRST-set predicate against a single
// candidate. Both callbacks receive the full candidateToken because
// omni's category predicates (isTypeFunctionName, isColId, isColLabel)
// consult BOTH the token type AND the literal string: `LookupKeyword(
// p.cur.Str)` must return a Keyword whose Token matches p.cur.Type. A
// predicateProbe that ignored candidateToken.name would make every
// UnreservedKeyword / TypeFuncNameKeyword probe fail the category check
// and report massive false drift.
//
// onNonKeyword may be nil to signal the production doesn't care about
// non-keyword starters — those candidates are skipped entirely in that
// case.
type predicateProbe struct {
    onKeyword    func(c candidateToken) bool
    onNonKeyword func(c candidateToken) bool // may be nil
}

// assertPredicateMatchesPG performs the full bidirectional FIRST-set check:
//
//   (1) PG accepts  ∧ predicate rejects → missingFromOmni (real drift; we
//       reject a token PG allows).
//   (2) predicate accepts ∧ PG rejects  → extraInOmni (predicate is too
//       permissive; commonly triggered by mis-categorized keywords that
//       enter isSimpleTypenameStart via the isTypeFunctionName fallback).
//
// Direction (2) is the check that catches mis-categorization drift: if
// bytea were wrongly marked TypeFuncNameKeyword, isSimpleTypenameStart
// would accept it via the fallback, PG would still reject `CAST(NULL AS
// bytea)` with 42601, and this direction flags the gap.
//
// Both directions iterate ALL outcomes returned by runFirstSetProbe, so
// non-keyword candidates (which collide on token==-1) are still each
// checked individually by display.
func assertPredicateMatchesPG(
    t *testing.T,
    production string,
    outcomes []probeOutcome,
    probe predicateProbe,
) {
    t.Helper()
    var missingFromOmni, extraInOmni []string

    for _, out := range outcomes {
        var predicateOK bool
        switch out.candidate.kind {
        case kindKeyword:
            predicateOK = probe.onKeyword(out.candidate)
        default:
            if probe.onNonKeyword == nil {
                continue // production doesn't care about non-keyword starters
            }
            predicateOK = probe.onNonKeyword(out.candidate)
        }
        switch {
        case out.accepted && !predicateOK:
            missingFromOmni = append(missingFromOmni, out.candidate.display)
        case !out.accepted && predicateOK:
            extraInOmni = append(extraInOmni, out.candidate.display)
        }
    }

    if len(missingFromOmni) > 0 || len(extraInOmni) > 0 {
        t.Errorf("%s FIRST-set drift:\n"+
            "  PG accepts but predicate rejects: %v\n"+
            "  predicate accepts but PG rejects: %v",
            production, missingFromOmni, extraInOmni)
    }
}
```

```go
func TestSimpleTypenameLeadTokensMatchPG(t *testing.T) {
    o := startFirstSetOracle(t)

    // Probe each keyword via CAST. CAST's grammar is
    //   CAST '(' a_expr AS Typename ')'
    // (gram.y:14130), which exercises `Typename`, a SUPERSET of
    // SimpleTypename: Typename = SimpleTypename opt_array_bounds
    //                          | SETOF SimpleTypename opt_array_bounds | ...
    //
    // For FIRST-set purposes the only difference is SETOF, since
    // opt_array_bounds is a suffix. We therefore subtract SETOF from
    // PG's accepted set below — the SimpleTypename production proper
    // does NOT include SETOF, so isSimpleTypenameStart must reject it,
    // but the CAST probe will accept `CAST(NULL AS setof int)` because
    // renderTypeCandidate produces that form.
    //
    // Result classification:
    //   42601 syntax_error → not a SimpleTypename lead
    //   42704 undefined_object / success → valid lead (unknown or built-in)
    //
    // Multi-token type syntax (DOUBLE PRECISION, NATIONAL CHARACTER,
    // CHARACTER VARYING, etc.) is covered by renderTypeCandidate trying
    // every rendering; the probe is accepted if any rendering parses.
    // Scope the probe to keyword + IDENT candidates. Including literals,
    // PARAM, and punctuation would only produce "both reject" matches
    // (PG rejects `CAST(NULL AS 1)`, predicate rejects int-typed tokens)
    // which is noise that could mask a real regression in the IDENT path.
    // filterKeywordsAndIdent keeps the probe focused on what SimpleTypename
    // actually discriminates on.
    filter := func(c candidateToken) bool {
        return c.kind == kindKeyword || c.kind == kindIdent
    }
    outcomes := runFirstSetProbe(t, o,
        "SELECT CAST(NULL AS %s)",
        renderTypeCandidate,
        filter,
    )

    // Subtract SETOF: CAST probes Typename, which includes SETOF, but
    // SimpleTypename does not. Without this subtraction the test would
```

### Task 1.5: Replace `isAExprConstTypeCast` with `isConstTypenameStart`

**Design note:** In omni's pg parser, the SimpleTypename and ConstTypename FIRST sets coincide at the hard-lead token level. They differ only in the category fallback: SimpleTypename falls through to `isTypeFunctionName` (GenericType path), ConstTypename does not. Rather than creating a second slice, we define `isConstTypenameStart` as a predicate that consults the **same** `simpleTypenameLeadSet` and simply omits the fallback. One source of truth for the lead tokens, two predicates for the two fallback policies.

**Why no PG oracle for ConstTypename specifically:** the one thing PG could validate that we care about — "does isConstTypenameStart reject tokens outside the shared lead set?" — is blocked by grammar ambiguity. PG's `AexprConst` position accepts both `ConstTypename Sconst` and `func_name Sconst`, and `func_name` starts on any IDENT / UnreservedKeyword / TypeFuncNameKeyword / ColNameKeyword, so `SELECT between '1'`, `SELECT text '1'`, `SELECT foo '1'` all parse via the `func_name` branch regardless of whether the lead token is in ConstTypename. The PG oracle would report almost every keyword as "accepted", giving us no signal. Since ConstTypename's lead set is identical to SimpleTypename's (which IS validated against PG), a PG oracle on ConstTypename is redundant anyway.

```go
// extractTopSwitchCases finds `func (p *Parser) funcName` and returns the
// set of identifier names used as case tags in its first TypeSwitch/Switch
// statement on p.cur.Type. Cases with non-identifier expressions fail the
// test.
func extractTopSwitchCases(t *testing.T, file *goast.File, funcName string) map[string]bool {
    t.Helper()
    var target *goast.FuncDecl
    for _, decl := range file.Decls {
        fn, ok := decl.(*goast.FuncDecl)
        if !ok {
            continue
        }
        if fn.Name.Name == funcName {
            target = fn
            break
        }
    }
    if target == nil {
        t.Fatalf("function %q not found", funcName)
    }

    var result map[string]bool
    goast.Inspect(target, func(n goast.Node) bool {
        sw, ok := n.(*goast.SwitchStmt)
        if !ok || result != nil {
            return true
        }
        // Check tag is p.cur.Type.
        sel, ok := sw.Tag.(*goast.SelectorExpr)
        if !ok || sel.Sel.Name != "Type" {
            return true
        }
        result = make(map[string]bool)
        for _, stmt := range sw.Body.List {
            cc := stmt.(*goast.CaseClause)
            for _, expr := range cc.List {
                id, ok := expr.(*goast.Ident)
                if !ok {
                    t.Fatalf("non-ident case in %s: %T", funcName, expr)
                }
                result[id.Name] = true
            }
        }
        return false
    })
    if result == nil {
        t.Fatalf("no switch on p.cur.Type found in %s", funcName)
    }
    return result
}
```

### Task 2.2: `isAExprStart` — coarse-grained expression lead check

**Grammar:** `a_expr` has a huge FIRST set (literals, identifiers, parens, operators, keywords like CASE / EXISTS / NULL, etc.). This is the most complex FIRST set in the grammar.

**Strategy:** rather than listing every token, define `isAExprStart` as the union of:

- `isConstTypenameStart()` (typed literals like `int '42'`, `interval '1 day'`)
- `p.isColId() || p.isTypeFunctionName()` — matches the existing idiom at `expr.go:1475` and `expr.go:2814`. omni's parser intentionally unions BOTH categories here because `isColId` covers UnreservedKeyword + ColNameKeyword while `isTypeFunctionName` adds TypeFuncNameKeyword; dropping the second category would reject valid expressions like `current_schema()` (CURRENT_SCHEMA is TypeFuncNameKeyword). **Do not replace with just `isColId`.**
- Literal tokens: `ICONST, FCONST, SCONST, BCONST, XCONST`, `NULL_P, TRUE_P, FALSE_P`
- Parameter reference: `PARAM` (for `$1` in prepared statements)
- Keyword openers: `CASE, EXISTS, NOT, LEAST, GREATEST, COALESCE, NULLIF, CAST, ARRAY, ROW, INTERVAL, CURRENT_TIMESTAMP, CURRENT_DATE, ...` (deliberately excluding `COLLATE`, which is a postfix operator in `a_expr COLLATE any_name`, not an expression starter)
- Punctuation openers: `'(', '+', '-'`

The slice is longer and less mechanically derivable than SimpleTypename. **Keep it conservative** — only add tokens we have a documented reason for, validated by the oracle test. The test uses `filterAll` (including PARAM / LPAREN / unary operators / IDENT) so a refactor that drops any of these is caught.

**Oracle probe template and filter:**

Bare `SELECT %s` is insufficient because many `a_expr` starters require follow-on syntax (`NOT true`, `EXISTS (SELECT 1)`, `CASE WHEN true THEN 1 END`, `CAST(NULL AS int)`, `ARRAY[1]`, `ROW(1)`). A bare probe would report these as syntax errors and push us to remove legitimate starters. Define a dedicated `renderExpression` function that expands each known expression-starter keyword into a valid SQL continuation:

```go
// renderExpression expands a candidate into plausible a_expr starts.
// Each expression-starter keyword needs a specific continuation because
// bare `SELECT not` / `SELECT exists` / `SELECT case` are all syntax errors.
//
// For tokens not in the expansion table, fall back to the bare form —
// most simple starters (IDENT, ICONST, SCONST, PARAM, '(1)', '+1', '-1',
// NULL, TRUE, FALSE, most type-name keywords) parse correctly bare.
//
// This table MUST cover every keyword that appears in the expected
// `isAExprStart` lead set and requires a continuation. Extend as Phase 2
// surfaces more.
func renderExpression(c candidateToken) []string {
    expansions := map[string][]string{
        "not":    {"NOT TRUE"},
        "exists": {"EXISTS (SELECT 1)"},
        "case":   {"CASE WHEN TRUE THEN 1 END"},
        "cast":   {"CAST(NULL AS int)"},
        "array":  {"ARRAY[1]"},
        "row":    {"ROW(1)"},
        // COLLATE is intentionally omitted: it is a POSTFIX operator
        // (a_expr COLLATE any_name), not a lead token. Including it here
        // would either misclassify it or require a rendering that starts
        // with a different token. Keep isAExprStart strict.
        "nullif": {"NULLIF(1, 2)"},
        "coalesce": {"COALESCE(1, 2)"},
        "greatest": {"GREATEST(1, 2)"},
        "least":  {"LEAST(1, 2)"},
        "interval": {"INTERVAL '1 day'"},
        "current_timestamp": {"CURRENT_TIMESTAMP"},
        "current_date":      {"CURRENT_DATE"},
        "current_time":      {"CURRENT_TIME"},
        "current_user":      {"CURRENT_USER"},
        "session_user":      {"SESSION_USER"},
        "user":              {"USER"},
        "localtime":         {"LOCALTIME"},
        "localtimestamp":    {"LOCALTIMESTAMP"},
        // NOTE: add more as Phase 2 uncovers them. Every added row should
        // have a comment or grammar reference explaining why the particular
        // continuation is the minimum valid form.
    }
    if c.kind == kindKeyword {
        if forms, ok := expansions[c.name]; ok {
            return forms
        }
    }
    return []string{c.probeSQL}
}

probe := "SELECT %s"
outcomes := runFirstSetProbe(t, o, probe, renderExpression, filterAll)
```

`filterAll` (not `filterKeywordsOnly`) is essential here — `a_expr`'s FIRST set genuinely includes `ICONST`, `SCONST`, `'('`, `'+'`, `'-'`, `PARAM` (for `$1`), bare `IDENT`, etc. The non-keyword seeds from `nonKeywordCandidates` (defined in Task 1.2) are exactly what's needed. This is also why the non-keyword seed set was introduced in Phase 1: so Phase 2 inherits a ready probe infrastructure and we don't have to retrofit the driver mid-phase.

**Phase 2 gating rule:** when `TestAExprLeadTokensMatchPG` first runs and shows PG-accepted tokens not in omni's expected lead set, triage each one. Legitimate expression starters that omni missed → add to `aExprLeadTokens`. False positives caused by renderExpression's continuations being too accommodating → tighten the expansion (e.g., a keyword that only parses because the continuation used it as an infix operator, not a lead) and re-run. Do NOT mass-accept or mass-reject — each drift entry is a human decision backed by a gram.y citation.

**Call site updates:** grep for `isColId() || isTypeFunctionName()` patterns in `expr.go` and `select.go`. These are probable `isAExprStart` candidates. Audit each — some are `column_ref` probes, not full expression probes, and should stay as `isColId`. Do not mass-replace.

```markdown
# PG Parser Maintenance Conventions

## FIRST-Set Discipline

omni's pg parser is recursive descent. It needs to answer "does this token start grammar production P?" for every lookahead/disambiguation point. These checks are called FIRST-set probes.

**Rule:** Every FIRST-set probe MUST go through a function named `isXStart(...)` backed by a single slice `xLeadTokens` defined in `first_sets.go`. No ad-hoc token case clusters are allowed in probe positions.

**Why:** Before 2026-04, FIRST sets were hand-duplicated across three call sites. One copy drifted, missing JSON and four other ColNameKeyword types, causing a customer-visible `CREATE FUNCTION f(data json)` failure. The single-source-of-truth rule prevents this class of drift.

**Enforcement:**
1. Every `xLeadTokens` slice is validated against real PG 17 by `TestXLeadTokensMatchPG` in `first_set_oracle_test.go`. Drift fails CI.
2. Every corresponding `parseX` dispatch switch has a `TestParseXSwitchMatchesLeadTokens` static check using `go/ast` that flags mismatches before the container spins up.
3. Code review: any PR introducing a `case TOK_A, TOK_B, TOK_C:` pattern in a non-dispatch context should be flagged as a FIRST-set duplication candidate.

## Oracle probe templates

Each production P has a probe template `probeP(name) string` that generates a minimal SQL statement exercising P at the target grammar position. The template must:

- Isolate P so that semantic errors on P (e.g., unknown type) are distinguishable from syntax errors outside P.
- Not embed the candidate in positions where other productions could match (e.g., do NOT use `SELECT x` as a SimpleTypename probe — it also matches column reference).
- Survive PG version upgrades — prefer stable constructs.

Current templates:

| Production | Template | Grammar ref |
|---|---|---|
| SimpleTypename | `SELECT CAST(NULL AS X)` | gram.y:14339 |
| ConstTypename | `SELECT X '1'` | gram.y:14370 |
| Typename | `CREATE FUNCTION __omni_probe() RETURNS X AS ...` | gram.y:14304 |
| func_type | `CREATE FUNCTION __omni_probe(p int) RETURNS X AS ...` | gram.y:8488 |
| a_expr | `SELECT X` | gram.y:14780 |

## Adding a new production

1. Define `xLeadTokens` in `first_sets.go` with a grammar reference comment.
2. Define `isXStart()` using `xLeadSet[p.cur.Type]` or composition with existing predicates.
3. Add probe template + `TestXLeadTokensMatchPG` in `first_set_oracle_test.go`.
4. (Optional) Add AST-based `TestParseXSwitchMatchesLeadTokens` for faster drift detection.
5. Replace call sites that probe FIRST of X with `isXStart()`. Grep for the old tokens to find candidates.
```

## Risks and open questions

1. **Oracle probe template subtleties.** Each probe SQL has to exercise the production we actually mean to test, not a containing production. Phase 1 uses `SELECT CAST(NULL AS X)` (true Typename/SimpleTypename position) rather than `CREATE FUNCTION ... RETURNS X` (which is parsed through func_type — a superset whose FIRST set only *coincidentally* equals Typename today). Phase 2's func_type test deliberately uses RETURNS so that any future divergence between `Typename` and `func_type` surfaces as a disagreement between the two oracle tests. Every new probe template must document: (a) which grammar production it exercises, (b) why the surrounding production doesn't pollute the FIRST-set measurement, (c) the gram.y line number of the rule being probed. Recorded in `pg/parser/CLAUDE.md` per Phase 4.

2. **Non-keyword starters need a seed list.** The oracle driver iterates `Keywords` plus a fixed `nonKeywordCandidates` list covering `IDENT` / numeric / string / bit / hex literals / `(` / unary `+` / unary `-`. Phase 1 tests filter to `kindKeyword` only — their FIRST sets are keyword-only at the hard-lead level, with IDENT coverage flowing through `isTypeFunctionName`. Phase 2's `isAExprStart` uses `filterAll`. If Phase 3 uncovers a production with a FIRST-set starter outside the seed list (e.g., `{`, `[`, JSONB-specific punctuation), extend `nonKeywordCandidates` rather than inventing a new candidate mechanism.

3. **Multi-token type syntax and per-candidate rendering.** Several PG type keywords only form valid types when combined with follow-on tokens (`DOUBLE` → `DOUBLE PRECISION`, `NATIONAL` → `NATIONAL CHARACTER`, `CHARACTER` → optionally `CHARACTER VARYING(n)`, etc.). A bare-substitution oracle would mark these as rejected and delete real lead tokens. The runner therefore calls a `renderFn` per candidate that enumerates plausible combinations (`base`, `base(1)`, `base precision`, `base character`, `base varying`, `base character varying(1)`, ...) and accepts the candidate if **any** rendering parses. The rendering list is derived from gram.y:14339..14760 and documented in CLAUDE.md so reviewers know why each form is there. If a new PG version adds a new multi-token type form, the renderer list needs one new entry — simple, localized, and the oracle test catches the miss.

4. **Bidirectional predicate validation.** The assertion helper iterates over every probed candidate, not just PG-accepted ones, and flags BOTH `PG accepts ∧ predicate rejects` (omni is missing a lead token) AND `predicate accepts ∧ PG rejects` (omni over-accepts, typically via a category fallback that now reaches a non-type token). The second direction is what catches mis-categorization drift — e.g. if `bytea` were wrongly marked `TypeFuncNameKeyword`, `isSimpleTypenameStart` would accept it via `isTypeFunctionName`, but PG would reject `CAST(NULL AS bytea)` with 42601, and direction (2) flags the gap. An earlier draft of the plan filtered categories out of "expected" before comparing, which would have hidden this class of drift. The current shape calls the predicate directly against PG's result set and walks all outcomes.

5. **Per-candidate outcome tracking.** The oracle runner returns `[]probeOutcome` (one entry per candidate), not a map keyed by token id. Non-keyword candidates (`IDENT`, `ICONST`, `(`, `+`, ...) all share `token == -1`, so any map keyed on `token` would collapse them into a single entry and make Phase 2's `isAExprStart` test unable to distinguish which non-keyword starter was accepted. The slice-of-outcomes shape lets the assertion walk each candidate individually by display.

2. **SQL state classification is coarse.** We currently map `42601 → reject` and everything else to `accept`. This works for the productions we've tested but may give false positives if PG returns e.g. `42P13 invalid_function_definition` for a token that IS a type but fails a separate check. Refine the classifier as we hit such cases; prefer a whitelist of accept codes over a blacklist of reject codes if the categorization proves noisy.

3. **Docker-less local dev vs misconfigured CI.** The helper inspects `$CI` and fails hard with `t.Fatalf` when setup errors occur under CI, but skips on local dev. This prevents the silent-disable failure mode where a misconfigured CI runner quietly skips every FIRST-set test and hides drift. Engineers without docker still get fast feedback via the `go/ast` static check from Task 1.6. The CI detection reads the standard `CI=true` variable (GitHub Actions, CircleCI, GitLab CI); if omni uses a different CI marker, substitute it.

4. **Token constant name reverse index.** Task 1.6's AST test needs to map `int` constants back to identifier names. If `tokens.go` uses a non-iota declaration pattern that breaks the parse, fall back to container-only validation for the affected production.

5. **PG version pinning.** The oracle pins `postgres:17-alpine` to match omni's parser target (see `pg/parser/type.go:9` and similar doc references throughout the package). Do NOT reuse `pg/catalog/container_helper_test.go`'s `postgres:16-alpine` image — PG 17 adds grammar (`JSON_TABLE` and friends, see `pg/parser/json.go`) that omni already parses, and a PG 17 oracle would flag those as "extra in omni" and incorrectly pressure their removal. When omni eventually upgrades its parser target to PG 18+, the FIRST-set oracle image bumps in the same PR; oracle tests flag grammar additions automatically and we update the slices. Document this upgrade flow in `pg/parser/CLAUDE.md` (Phase 4).

6. **Phase 2 `isAExprStart` scope creep.** `a_expr` has hundreds of productions. It's tempting to make `isAExprStart` "the complete expression FIRST set" but that leads to a massive hand-maintained list. Alternative: treat `isAExprStart` as "expression-like enough to start parsing" and rely on `parseAExpr` to reject bad continuations. Decide in Phase 2 before implementing.
